云提供商故障引起的反思

发表于 2016-12-09 | 分类于 saltstack | | 阅读次数

背景介绍

来公司的时候，公司已经使用了云提供商。接手的项目是一个经验很少的运维工程师搭建的服务器，没有使用自动化管理软件。
ulb是云厂商提供的loadbalance软件，作为基础服务。由于目前国内云厂家提供的云服务不是很稳定，已经做得是梳理云端单点，
能使用ulb的地方也都使用了。但是由于使用的镜像，唯独没有对dns进行统一管理，一直配置在/etc/sysconfig/network-scripts/ifcfg-eth0里面。

故障

12月8日凌晨两点二十多，手机开始收到zabbix服务器unreachable的告警。然后我就起床了，这也是感冒加重了。我先ping了下zabbix的域名，网络不通了。
于是ping ulb发现已经不通了。然后看下dns服务器，发现服务正常。已经确定是云厂商的ulb挂了。启用备份机制，将后端的dns发给各位兄弟们，让大家改配置。
云厂家吹得再牛逼，也不要相信了，这tm都是线上呀，不能抱着侥幸心理呀。我就是犯了这个错误。虽然云四十多分钟之后恢复了，但是这让我更加意识到统一平台的
重要性。手动改网络配置累死吧。

后续改造

所有的能自动化做的，也不通过镜像做了，将ifcfg-eth0中的dns删除，直接配置/etc/resolv.conf配置文件。批量分发到需要使用dns服务器的客户端上，再出现故障争取
分钟级恢复。不要存在懒惰心里，会死得很惨的。虽然第二天你可以把责任推脱给供应商，但是我们也要做到自己心目中的唯一。闲谈不议它非，静坐常思己过。

下面将从如下几个方面来讲述deployment：

方法一：
kubectl set image deployment/nginx-deployment nginx=nginx:1.7.9

方法二：
kubectl edit deployment/nginx-deployment

通过这两个方法修改之后，可以查看当前的情况信息：
$ kubectl get deployments

UP-TO-DATE: 表明deployment已经读取了最新的配置信息中rc的个数；
current：当前deployment管理的rc个数；
available：表明当前rc中那些副本是可用的；

CMD和ENTRYPOINT指令

CMD和ENTRYPOINT指令都非常简单，但它们都有一个隐藏的容易出错的“功能”，
如果你不知道的话可能会在这里踩坑，这些指令支持两种不同的语法。
CMD /bin/echo
＃or
CMD [“/bin/echo”]
这看起来好像没什么问题，但仔细一看其实两种方式差距很大。如果你使用第二个语法：CMD（或ENTRYPOINT）是一个数组，
它执行的命令完全像你期望的那样。如果使用第一种语法，Docker会在你的命令前面加上/bin/sh -c，我记得一直都是这样。
如果你不知道Docker修改了CMD命令，在命令前加上/bin/sh -c可能会导致一些意想不到的问题以及难以理解的功能。
因此，在使用这两个指令时你应当使用数组语法，因为数组语法会确切地执行你打算执行的命令。

CMD和ENTRYPOINT 结合使用更好

docker run命令中的参数都会传递给ENTRYPOINT指令，而不用担心它被覆盖（跟CMD不同）。当与CMD一起使用时ENTRYPOINT的表现会更好。
