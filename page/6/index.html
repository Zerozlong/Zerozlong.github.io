k8s新版本安装（1.6.2）

发表于 2017-05-11 | 分类于 k8 | | 阅读次数

国内镜像源配置

docker.repo
cat >> /etc/yum.repos.d/docker.repo <<EOF
[docker-repo]
name=Docker Repository
baseurl=http://mirrors.aliyun.com/docker-engine/yum/repo/main/centos/$releasever/
enabled=1
gpgcheck=0
EOF

kubernetes yum源
cat >> /etc/yum.repos.d/kubernetes.repo <<EOF
[kubernetes]
name=Kubernetes
baseurl=https://mirrors.aliyun.com/kubernetes/yum/repos/kubernetes-el7-x86_64/
enabled=1
gpgcheck=0
EOF

安装docker

查看docker版本
yum list docker-engine –showduplicates

Kubernetes 1.6还没有针对docker 1.13和最新的docker 17.03上做测试和验证，所以这里安装Kubernetes官方推荐的Docker 1.12版本。
yum install -y docker-engine-1.12.6-1.el7.centos.x86_64

根据官方文档Installing Kubernetes on Linux with kubeadm中的Limitations小节中的内容，对各节点系统做如下设置:

flannel网络踩坑续

具体体现在使用cni网络之后，pod的网络可以跨主机与其它主机的pod通信；但是网络却不能与其它主机的主机ip通信。
鉴于cni网络的各种bug，现决定使用flannel老版本。

在etcd集群上创建一个pod网段，使用如下命令：

etcdctl mkdir /coreos.com/network
etcdctl set /coreos.com/network/config ‘{“Network”:”172.17.0.0/16”}’

由于 Etcd 和 Kubernetes 全部采用 TLS(传输层安全通信协议) 通讯，所以先要生成 TLS 证书，证书生成工具采用 cfssl，
具体使用方法这里不再详细阐述，生成证书时可在任一节点完成，这里在宿主机执行，证书列表如下:
etcd根ca证书：ca.pem
etcd 集群证书：kubernetes-key.pem、kubernetes.pem；
kube-apiserver：使用 ca.pem、kubernetes-key.pem、kubernetes.pem；
kubelet：使用 ca.pem；
kube-proxy：使用 ca.pem、kube-proxy-key.pem、kube-proxy.pem；
kubectl：使用 ca.pem、admin-key.pem、admin.pem；
‘’’

证书生成

‘’’
wget https://pkg.cfssl.org/R1.2/cfssl_linux-amd64
wget https://pkg.cfssl.org/R1.2/cfssljson_linux-amd64
chmod +x cfssl_linux-amd64 cfssljson_linux-amd64
mv cfssl_linux-amd64 /usr/local/bin/cfssl
mv cfssljson_linux-amd64 /usr/local/bin/cfssljson
cd /export/kubernetes/ssl
cfssl print-defaults config > config.json
cfssl print-defaults csr > csr.json
手动添加ca配置文件：
[root@JXQ-23-58-39 ssl]# cat ca-config.json
{
“signing”: {
“default”: {
“expiry”: “87600h”
},
“profiles”: {
“kubernetes”: {
“usages”: [
“signing”,
“key encipherment”,
“server auth”,
“client auth”
],
],
“expiry”: “87600h”
}
}
}
}
字段说明：

  • ca-config.json：可以定义多个 profiles，分别指定不同的过期时间、使用场景等参数；后续在签名证书时使用某个 profile；
    signing：表示该证书可用于签名其它证书；生成的 ca.pem 证书中 CA=TRUE； server auth：表示client可以用该 CA 对server提供的证书进行验证；
    *client auth：表示server可以用该CA对client提供的证书进行验证；
    手动创建CA证书签名请求：
    cat >> ca-csr.json <<EOF
    {
    “CN”: “kubernetes”,
    “key”: {
    “algo”: “rsa”,
    “size”: 2048
    },
    “names”: [
    {
    "C": "CN",
"ST": "BeiJing",
"L": "BeiJing",
"O": "k8s",
"OU": "System"
    }
    ]
    }
    EOF
  • “CN”：Common Name，kube-apiserver 从证书中提取该字段作为请求的用户名 (User Name)；浏览器使用该字段验证网站是否合法；
  • “O”：Organization，kube-apiserver 从证书中提取该字段作为请求用户所属的组 (Group)；

创建kubernetes证书：

cat >> kubernetes-csr.json <<EOF
{

],

},

{
“C”: “CN”,

注意：如果 hosts 字段不为空则需要指定授权使用该证书的 IP 或域名列表，
由于该证书后续被 etcd 集群和 kubernetes master 集群使用，所以上面分别指定了 etcd 集群、kubernetes master 集群的主机 IP ,以及后续的service ip地址范围的第一个
ip地址；如 10.254.0.1
生成kubernetes和etcd的证书：
cfssl gencert -ca=ca.pem -ca-key=ca-key.pem -config=ca-config.json -profile=kubernetes kubernetes-csr.json | cfssljson -bare kubernetes

创建 admin 证书

创建admin证书签名请求：
cat >> admin-csr.json <<EOF
{
“CN”: “admin”,
“hosts”: [],
“key”: {
“algo”: “rsa”,
“size”: 2048
},
“names”: [
{
“C”: “CN”,
“ST”: “BeiJing”,
“L”: “BeiJing”,
“O”: “system:masters”,
“OU”: “System”
}
]
}
EOF

  • 后续 kube-apiserver 使用 RBAC 对客户端(如 kubelet、kube-proxy、Pod)请求进行授权；
  • kube-apiserver 预定义了一些 RBAC 使用的 RoleBindings，如 cluster-admin 将 Group system:masters 与 Role cluster-admin 绑定，该 Role 授予了调用kube-apiserver 的所有 API的权限；
  • OU 指定该证书的 Group 为 system:masters，kubelet 使用该证书访问 kube-apiserver 时 ，由于证书被 CA 签名，所以认证通过，同时由于证书用户组为经过预授权的 system:masters，
    所以被授予访问所有 API 的权限；
    生成 admin 证书和私钥
    cfssl gencert -ca=ca.pem -ca-key=ca-key.pem -config=ca-config.json -profile=kubernetes admin-csr.json | cfssljson -bare admin

glusterfs使用nfs方式

发表于 2017-04-19 | 分类于 分布式存储 | | 阅读次数

什么情况下用nfs方式？

不想让某个目录暴漏在所有的客户端，使用glusterfs做好目录划分：
mkdir /test
mount -t glusterfs service1:/test-volume /test
cd /test && mkdir -v project{1,2,3,4,5,6}
这个时候我们想让project1只暴露给project1的主机；

怎么样用glusterfs的nfs协议？

首先打开逻辑卷的nfs协议；
mount -t nfs -o vers=3 server1:/test-volume/project1 /mnt/glusterfs

注：Gluster NFS server不支持UDP，如果你的NFS客户端报错提示NFS版本或协议不正确，则可以在挂接命令的参数选项中明确指定下使用TCP协议。
mount -o mountproto=tcp -t nfs server1:/test-volume /mnt/glusterfs

设置成系统启动时自动挂接存储卷

设置成系统启动时自动挂接存储卷：
修改/etc/fstab文件并添加以下内容：
server1:/test-volume /mnt/glusterfs nfs defaults,_netdev,vers=3 0 0
或
server1:/test-volume /mnt/glusterfs nfs defaults,_netdev,mountproto=tcp 0 0

续言

自己刚开始学习代码的时候，看书上的装饰器感觉理解了，就照着装饰器的样子写代码。一段时间之后，好像又陌生了。只知道这样写，能实现这样的功能。然后没放在
心上。直到某天我在写类的时候，里面也出现了好多装饰器，我觉得是时候从理论上再来一遍学习了。

装饰器是什么？

书上说的不是太清楚，根据我的理解就是类似于程序设计中你要在其前面和后面执行什么操作。当然你也可以做更多的事情，如：引入日志、增加计时逻辑来检测性能、给函数增加
事务的能力。
小结：装饰器其实也就是一个函数，一个用来包装函数的函数，返回一个修改之后的函数对象，将其重新赋值之后。使用原来的标识符，并永久丧失对原始函数对象的访问。

注意：Python的函数具有非常灵活的参数形态，既可以实现简单的调用，又可以传入非常复杂的参数。默认参数一定要用不可变对象，如果是可变对象，运行会有逻辑错误！要注意定
义可变参数和关键字参数的语法：args是可变参数，args接收的是一个tuple；**kw是关键字参数，kw接收的是一个dict。以及调用函数时如何传入可变参数和关键字参数的语法：
可变参数既可以直接传入：func(1, 2, 3)，又可以先组装list或tuple，再通过args传入：func((1, 2, 3))；关键字参数既可以直接传入：func(a=1, b=2)，又可以先组装dict，
再通过kw传入：func(**{‘a’: 1, ‘b’: 2})。使用args和**kw是Python的习惯写法，当然也可以用其他参数名，但最好使用习惯用法。
